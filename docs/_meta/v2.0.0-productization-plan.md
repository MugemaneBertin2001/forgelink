# ForgeLink v2.0.0 — Productization Plan

**Date prepared:** 2026-04-22
**Target release:** v2.0.0 (no earlier than 2026-10-14; six months out)
**Preceding release:** v1.1.0 (six weeks out; public-worthy demo checkpoint)
**Budget assumption:** 60 hours/month solo, degrading to 40 hours/month under day-job + Upwork + Copa pressure.

## 0. Framing and version name

The retrospective diagnosed v1.0.0 honestly: a scaffold with multiple subsystems deployed-but-not-wired. Honest labelling was the right v1.0 answer; it is the wrong v2.0 answer. The bar for v2.0.0 is that every claim in the README, in the architecture docs, and in consulting positioning is backed by running behavior a reviewer can observe within ten minutes of encountering the project.

**Version name: v2.0.0 (not v1.2.0 or v1.5.0).** Justified because this cycle introduces breaking changes:

- NetworkPolicy allow-list enforcement breaks any deploy workflow that assumed flat intra-namespace reachability.
- Service-to-service mTLS (M2) breaks clients that do not present a cert.
- The delete list removes features (SPIRE claim, RabbitMQ, MinIO, `apps/ai/`, `slack-bot/`, HashiCorp Vault references) that downstream tooling may have depended on.
- The feature-surface semantics shift materially enough that semver discipline demands the major bump.

### SPIRE decision — DELETE

The v1.0.0 retrospective and zero-trust.md both acknowledge SPIRE is deployed-not-wired. Wiring one workload (Spring IDP) honestly takes 2–3 weeks of focused work; wiring three runtimes (Python, Java, Dart) for full SVID consumption is 6–10 weeks and introduces lifecycle complexity (trust domain bootstrap, CRD management, attestor validation) that has no demo-visible payoff for the consulting audience ForgeLink targets.

**Replacement:** Service-to-service mTLS via cert-manager-issued certs from an internal ClusterIssuer (CA type). This delivers the wire-level security the SPIRE claim promised, within the 60 hr/month budget, with evidence a reviewer can see (Spring access logs showing client DN; handshake failures from cleartext probes).

**User's "Zero-Trust-Lite" opt-out (from the planning brief) is explicitly taken here.** The README ships with "JWT (RS256) + Permission-based RBAC + NetworkPolicy allow-list + cert-manager-issued mTLS" as the security claim. No SPIFFE, no SPIRE. The zero-trust doc in v2.0.0 has no SPIRE sections.

This is reversible. If a future consulting engagement requires SPIFFE/SPIRE specifically, we reintroduce in v3.0.0.

### NetworkPolicy allow-list (M1)

- **Acceptance:** Every intra-cluster communication path used by the demo is explicitly allow-listed; any other pair is denied. `kubectl exec` into an unauthorized probe pod + curl a protected service → timeout. `kubectl logs` on CNI shows the denial.
- **Work:** 8–10 NetworkPolicy manifests under `k8s/base/network-policies/allow-list/` (one per tier: api↔idp, api↔postgres, api↔redis, api↔kafka, api↔tdengine, api↔emqx, notification↔kafka, notification→egress-smtp-slack, ingress→api+idp, spire-ns-internal-if-kept). Test matrix: for each manifest, confirm allow + deny with a netshoot pod. k3s install must have Calico or equivalent NetworkPolicy-aware CNI (default flannel does not enforce NetworkPolicy).
- **Evidence:** `kubectl get networkpolicy -n forgelink` shows 10+ policies; a committed test script (`scripts/test-networkpolicy.sh`) exercises the matrix and outputs pass/fail; zero-trust.md v2 includes captured logs.
- **Dependencies:** Calico CNI on k3s (install-time flag or add-on).
- **Effort:** 5 days (2 to write, 3 to test thoroughly).

## 3. Demo plan

### Hosting target — OCI Always Free, resource budget

OCI Always Free ARM limits (per account):
- **Compute:** up to 4 OCPU + 24 GB RAM across A1.Flex instances.
- **Block storage:** 200 GB total.
- **Load Balancer:** 1 flexible (10 Mbps) free.
- **Object Storage:** 20 GB free.
- **Networking:** 10 TB egress free per month.

Per-service resource budget for the full stack:

| Service | CPU req | Mem req | Notes |
|---|---|---|---|
| k3s control plane overhead | 0.50 | 1.0 GB | Single-node cluster |
| PostgreSQL | 0.25 | 1.0 GB | Shared across forgelink + idp schemas |
| Redis | 0.10 | 256 MB | Low memory ceiling |
| Kafka (KRaft) | 0.50 | 2.0 GB | JVM heap tuned low |
| TDengine | 0.50 | 2.0 GB | |
| EMQX | 0.25 | 1.0 GB | |
| Django API (Gunicorn) | 0.50 | 1.0 GB | 2 workers |
| Celery worker | 0.25 | 512 MB | 1 worker in demo |
| Celery beat | 0.10 | 256 MB | |
| Kafka consumer | 0.25 | 512 MB | |
| Spring IDP | 0.25 | 768 MB | JVM heap 512 MB |
| Spring Notification | 0.10 | 512 MB | |
| OPC-UA simulator | 0.10 | 256 MB | |
| Edge gateway | 0.10 | 256 MB | |
| MQTT bridge | 0.10 | 256 MB | |
| Prometheus | 0.25 | 1.0 GB | Short retention |
| Grafana | 0.10 | 256 MB | |
| cert-manager | 0.10 | 256 MB | |
| **Total requests** | **4.30** | **13.1 GB** | |

CPU request totals (4.30) exceed the 4 OCPU ceiling by 7.5%. Mitigations:

1. **Oversubscription is fine** — CPU requests are guaranteed minimums under contention; typical utilization is <20%. Kubernetes will schedule this.
2. **Split to 2× A1.Flex instances** if CPU contention materializes — free-tier allows that total (4 OCPU) split across multiple A1 instances. Promote one instance to worker; scheduler spreads pods. Acts as a fallback without leaving the free tier.
3. **Deferred-optional services** (Prometheus short-retention, Celery beat single-replica) stay minimal.

Memory at 13.1 GB fits 24 GB ceiling with 10 GB headroom — comfortable.

Block storage: ~60 GB for TDengine + Postgres + Kafka + Prometheus logs. Well under 200 GB.

**Verdict:** Full stack fits OCI Always Free with minor oversubscription. If we hit CPU walls during load, split to 2× A1 instances (still free). No service moves to paid tier.

## 5. Branching strategy

**Hybrid: short-lived topic branches off main, tags cut from main, no long-lived feature branch.**

Justification:
- `main` must remain releasable. A six-month feature branch accrues merge conflicts a solo engineer cannot afford.
- Each work item in Section 2 is independently shippable: cert-manager alone is a complete change; NetworkPolicy allow-list is a complete change.
- Topic branches have 3–7 day lifetimes. Each merges to main after CI green + manual smoke test on the OCI demo (which redeploys from main automatically once CD lands).

Topic branch names:
- `feat/scope-cut` — the delete list, single PR
- `feat/cert-manager` — M1
- `feat/networkpolicy-allowlist` — M1
- `feat/observability-k8s` — M1
- `feat/oci-demo-deploy` — M1
- `feat/mobile-apk-release` — M1
- `feat/emqx-tls` — M2
- `feat/mtls-internal-pki` — M2
- `feat/notification-email` — M2
- `feat/performance-characterization` — M3
- `feat/runbooks-and-adrs` — M3
- `feat/demo-polish` — M4
- `chore/rewrite-docs-present-tense` — M4

Exception: the delete-list branch touches many files and benefits from a single careful review pass. Kept small by being purely deletion.

Tags cut from main at M1 completion (v1.1.0) and M4 completion (v2.0.0). M2 and M3 do not tag — they land on main as-they-go.
