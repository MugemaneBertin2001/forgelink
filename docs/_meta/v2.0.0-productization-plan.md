# ForgeLink v2.0.0 — Productization Plan

**Date prepared:** 2026-04-22
**Target release:** v2.0.0 (no earlier than 2026-10-14; six months out)
**Preceding release:** v1.1.0 (six weeks out; public-worthy demo checkpoint)
**Budget assumption:** 60 hours/month solo, degrading to 40 hours/month under day-job + Upwork + Copa pressure.

## 0. Framing and version name

The retrospective diagnosed v1.0.0 honestly: a scaffold with multiple subsystems deployed-but-not-wired. Honest labelling was the right v1.0 answer; it is the wrong v2.0 answer. The bar for v2.0.0 is that every claim in the README, in the architecture docs, and in consulting positioning is backed by running behavior a reviewer can observe within ten minutes of encountering the project.

**Version name: v2.0.0 (not v1.2.0 or v1.5.0).** Justified because this cycle introduces breaking changes:

- NetworkPolicy allow-list enforcement breaks any deploy workflow that assumed flat intra-namespace reachability.
- Service-to-service mTLS (M2) breaks clients that do not present a cert.
- The delete list removes features (SPIRE claim, RabbitMQ, MinIO, `apps/ai/`, `slack-bot/`, HashiCorp Vault references) that downstream tooling may have depended on.
- The feature-surface semantics shift materially enough that semver discipline demands the major bump.

### SPIRE decision — DELETE

The v1.0.0 retrospective and zero-trust.md both acknowledge SPIRE is deployed-not-wired. Wiring one workload (Spring IDP) honestly takes 2–3 weeks of focused work; wiring three runtimes (Python, Java, Dart) for full SVID consumption is 6–10 weeks and introduces lifecycle complexity (trust domain bootstrap, CRD management, attestor validation) that has no demo-visible enforcement signal for the consulting audience ForgeLink targets.

**Replacement:** Service-to-service mTLS via cert-manager-issued workload certs from an internal ClusterIssuer (CA type). Equivalent mTLS posture with observable handshake evidence (Spring access logs show client DN; cleartext probes fail at handshake). Achievable within the 60 hr/month budget.

**User's "Zero-Trust-Lite" opt-out (from the planning brief) is explicitly taken here.** The README ships with "JWT (RS256) + Permission-based RBAC + NetworkPolicy allow-list + cert-manager-issued mTLS" as the security claim. No SPIFFE, no SPIRE.

This is reversible in v3.0.0 if a future engagement demands SPIFFE specifically.

**Execution rule:** `docs/architecture/zero-trust.md` is rewritten in the same commit that removes the SPIRE manifests (not a follow-up commit). The commit message reads `feat: delete SPIRE manifests, switch zero-trust doc to cert-manager PKI`. This prevents a window where the manifests are gone but the doc still references them.

OCI Always Free ceiling: 4 OCPU + 24 GB RAM total.

**CPU headroom analysis:**
- Requests (4.30) exceed the 4 OCPU ceiling by 7.5%. This is sustainable because Kubernetes requests are guaranteed floors; actual steady-state utilization on this workload is ~15–20% aggregate.
- Limit totals (13.75) are nominal burst capacity; the scheduler will oversubscribe aggressively. Under genuine concurrent bursts, CPU throttling occurs in that order: Prometheus → Celery → Django. Acceptable — none of those are demo-critical-path.

**RAM headroom analysis:**
- Requests (13.1 GB) leave 10.9 GB below the 24 GB ceiling. Comfortable at steady state.
- Limits (20.1 GB) leave 3.9 GB of the 24 GB ceiling for kernel, systemd, k3s overhead, and burst headroom. Tight but above the user-specified 20 GB threshold for the "OCI-only" verdict.
- Bursty services (Kafka, TDengine, Postgres, Prometheus) account for 11.5 GB of the 20.1 GB limit total — if two burst concurrently at max limit (e.g., a telemetry spike pushing Kafka + TDengine together), the cluster still has headroom, but memory pressure is the first warning signal.

**Verdict:** Full stack fits OCI Always Free at requests. At limits, we are within the ceiling but with the warning that simultaneous max-burst across all four bursty services is not accommodated. This is acceptable for a steady demo workload but not for a sustained load test.

**Local-only fallback (if OCI budget becomes insufficient):**

If a future scope addition (e.g., Loki in v2.1.0, or OTel in v2.2.0) pushes totals past 20 GB, the following services move to `docker-compose.yml` only (for reviewer clone-and-run) and are not OCI-resident:

1. **Prometheus + Grafana** — lowest demo-value-per-GB. A reviewer can point their browser at local Grafana when cloning; the hosted demo continues serving the core UI without observability.
2. **cert-manager** — moves to manual cert issuance on the hosted demo if needed (retain on local).
3. **Last resort — opcua-simulator + edge-gateway** — run only in local docker-compose mode, with the hosted demo using a pre-seeded static TDengine snapshot. This removes live telemetry from the demo and is the last fallback before cutting paid-tier compute.

No service moves to OCI paid tier without explicit scope approval.

**Load-test budget note:** Performance characterization in M3 runs on a separate OCI VM (can be paid tier for a one-week window, estimated ~$15–30 one-time cost) so the public demo is not impacted. Demo continues on Always Free throughout.

- Topic branches have 3–7 day lifetimes. Each merges to main after CI green + manual smoke test on the OCI demo (which redeploys from main automatically once CD lands).

### "Releasable on main" — the enforceable checklist

The constraint "main must remain releasable" is enforced, not aspirational. Before merging any topic branch, every item below must be true on the resulting main state. A pre-merge smoke check runs through this list:

1. **`docker compose up -d` completes and stays healthy.** All advertised services reach their healthcheck green state within 3 minutes on a clean laptop. No CrashLoopBackOff, no dependency failures.
2. **End-to-end smoke holds.** Seed the simulator, POST a test telemetry record, verify it lands in TDengine via the Kafka consumer path. Trigger a test alert, verify it hits Slack (or the demo sink).
3. **The v1.0.0 git tag still builds.** `git checkout v1.0.0 && docker compose build` succeeds. A regression in the base image or a dropped dep that breaks historical tags is a main-state regression.
4. **No broken internal links in docs.** Run the link-audit check (`scripts/check-doc-links.sh` — to be committed in M1 week 1 if not yet present). Every internal markdown link resolves.
5. **No half-landed feature branches.** Every merged branch must be complete against its own acceptance criteria; partial work is committed on its topic branch, never merged to main.
6. **CI green on the merge commit.** All four workflows (`ci`, `build`, `deploy` dry-run, `release` dry-run) pass.
7. **`scripts/deploy.sh` to a fresh local k3s completes successfully.** Demonstrates main is cluster-deployable, not just compose-runnable.

A merge that breaks any of these is rolled back with `git revert` on main; the topic branch is reopened for repair. No "forward-fix it next PR" on main.

This checklist is codified (not just described here) as `scripts/check-releasable.sh` in M1 week 1. CI runs it as a required check on merge-to-main.

Tags cut from main at M1 completion (v1.1.0) and M4 completion (v2.0.0). M2 and M3 do not tag — they land on main as-they-go.

## Appendix A — Clarifications resolved during plan review

Persistent record of the three clarifying questions raised during plan review and their evidence-based answers, so future readers of the plan do not have to reconstruct the reasoning from chat history.

**A.1 `mobile/flutter-app/` delete scope.** Verified as legacy scaffolding, not scope. `find mobile -type f` returns a single file (`mobile/flutter-app/README.md`); the `lib/` subdirectory is empty. The real Flutter app is `services/flutter-app/` (21 `.dart` files, Riverpod + GoRouter + Dio + Socket.IO). The `mobile/` tree was the original scaffold location; the active app was moved to `services/` during phase 3 and the original tree was never cleaned up. Delete is safe.

**A.2 RabbitMQ delete — architectural defense.** The original CLAUDE.md scoped RabbitMQ for "commands and alerts." The as-built system delivers both roles through other transports:
- **Commands** flow through MQTT on a per-area `commands` subtree, enforced by the EMQX ACL (`config/emqx/acl.conf:25,31,37,43`). Per-device request-reply semantics come from MQTT QoS 1 + the device's own subscription on its command subtree.
- **Alert dispatch** runs on Kafka via the `alerts.notifications` topic, consumed by the Spring Notification Service.
- **Celery** (for internal async work) uses Redis as broker (`CELERY_BROKER_URL = "redis://..."` in `apps/core/settings.py:249`), not RabbitMQ.
- No AMQP client is imported anywhere in `services/` (grep for `pika` / `amqp` returns zero hits against code).

Deleting RabbitMQ is a drift correction to match the as-built architecture, not a scope cut. If a future feature requires AMQP-specific semantics (topic exchanges, routing keys for command orchestration), we reintroduce it in that feature's release.

**A.3 OPC-UA simulator + Edge Gateway — KEEP, demo-critical.** These are the demo's data source. Together they produce the continuous telemetry that populates every Grafana panel, every device detail page, and every alert trigger on the live demo. Demo-visible function:
- **OPC-UA simulator** — exposes an OPC-UA address space mirroring the ISA-95 steel-plant hierarchy, driven by Django Simulator app via Redis pub/sub.
- **Edge Gateway** — OPC-UA client that subscribes to simulator value changes and republishes onto the MQTT UNS topics.

Without both, the demo dashboard is empty. Both also serve as the credibility artefact for the consulting claim "we understand the OT edge" — a reviewer can inspect the OPC-UA address space and see it modelling a plant, not a toy dataset. Both run on the OCI demo 24/7.
